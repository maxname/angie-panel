## angie-facts

**Вердикт/резюме:** План фактологически на удивление точен в ядре: выбор Angie >= 1.11 корректен и даже необходим (и acme_http_port, и /status/http/acme_clients/ появились именно в 1.11.0, 24.12.2025; актуальный релиз — 1.11.8 от 18.06.2026); подтверждены синтаксис и дефолты acme_client (challenge=http, key_type=ecdsa, renew_before_expiry=30d, retry_after_error=2h), контекст http (включая include-файлы), переменные $acme_cert_*/$acme_cert_key_*, хранилище /var/lib/angie/acme/<client>/ (account.key, certificate.pem, private.key), «один клиент = один сертификат на все server_name ссылающихся серверов», пропуск regex-имён, wildcard только через dns-01, обязательный resolver (127.0.0.53 используется в официальных примерах), автооткат при reload, http2 on, arm64-пакеты для Debian 11/12/13 с вкомпилированным --with-http_acme_module, наличие angie-console-light, отсутствие write-API в OSS (динамический /config — только в PRO), NS-делегирование только на порт 53 (свойство протокола DNS) и открытие UDP-сокета acme_dns_port master-процессом (ngx_create_listening на этапе загрузки конфига). Однако найдены две структурные ошибки: (1) пакетный angie.conf включает только /etc/angie/http.d/*.conf — подкаталог hosts/ из раскладки §4 не загрузится вообще; (2) заявление «angie -t от пользователя панели с -e stderr не требует прав» неверно — в тест-режиме Angie открывает все error_log/access_log из конфига, создаёт pid-файл и, главное, при наличии acme_client создаёт каталоги 0700 и генерирует ключи в /var/lib/angie/acme — под непривилегированным пользователем тест гарантированно падает. Плюс план не учитывает пакетный /etc/angie/http.d/default.conf (сервер на :80 с /status API), закомментированный stream-блок в пакетном конфиге, скудность полей /status/http/acme_clients/ (нет даты истечения и списка доменов) и упускает ряд возможностей Angie (challenge=alpn, api_config_files, ssl_reject_handshake, renew_on_load, Prometheus-модуль). Ненужное наследие NPM: исключение /.well-known/acme-challenge/ из force-ssl-редиректа не требуется — модуль перехватывает челленджи на фазе POST_READ до выполнения return/rewrite.

## rust-stack

**Вердикт/резюме:** Архитектура в целом грамотная и заметно лучше среднего для таких панелей: выбор Rust-стека (axum/tokio/sqlx+SQLite bundled, askama, rust-embed, tower-sessions, utoipa, hickory-resolver, zbus, argon2id) технически корректен и весь стек собирается в musl-статик под aarch64; факты про Angie (acme_http_port, $acme_cert_* переменные, /status/angie.generation, /status/http/acme_clients, обязательный resolver, «один клиент = один сертификат», /var/lib/angie/acme) проверены по документации и в плане изложены верно; пайплайн staging→diff→backup→test→reload→verify концептуально правильный. Однако два элемента фундамента сломаются на практике в текущем виде: (1) «angie -t -e stderr» от непривилегированного пользователя не работает — -e подменяет только дефолтный лог, а тест всё равно открывает error_log/access_log из конфигов и создаёт pid-файл (это надо перепроектировать на root-исполняемый oneshot-юнит через тот же polkit либо точный sudoers, а не оставлять «проверить на M0»); (2) атомарный rename из staging в /etc/angie/http.d даст EXDEV под заявленным ProtectSystem=strict, потому что ReadWritePaths — это отдельные bind-mount'ы (temp-файлы надо создавать внутри целевого каталога). Кроме того: подкаталог hosts/ не попадёт в пакетный include *.conf; polkit-правило с unit/verb-скоупингом работает только на polkit ≥0.106 (Debian 12/13 — да, Debian 11 из install.sh — нет) и требует Depends: dbus/polkitd; sudo-фолбэк несовместим с NoNewPrivileges-implying харденингом; права на http.d нужно фиксировать через dpkg-statoverride, а не chmod в postinst; для показа причины неудачного reload панели нужен доступ к error.log (angie -t не ловит конфликты портов, т.к. не биндит сокеты); в rustls следует явно закрепить ring-провайдер вместо дефолтного aws-lc-rs для беспроблемного musl-кросса. С этими исправлениями (все — локальные, без смены архитектуры) модель привилегий и пайплайн становятся реалистичными; отдельно стоит честно задокументировать, что запись в http.d по сути даёт root-эквивалент через директивы Angie, так что непривилегированность панели — защита от её багов, а не от её компрометации.

**Детали:**
- [critical] (§2.1 / §2.2 / §11 — 'angie -t' от непривилегированного пользователя) Предположение «angie -t с -e stderr не требует прав» неверно, и это сломается в первый же день, а не как edge case. '-e stderr' (унаследован от nginx 1.19.5, в Angie есть) подменяет только ВСТРОЕННЫЙ дефолтный error log. ## completeness

**Вердикт/резюме:** This is an unusually strong plan for its stage: the apply pipeline (staging/diff/snapshot/rollback/verified reload), no-default-credentials setup, non-root privilege model, and native Angie ACME directly attack NPM's four worst pain points, and the M1-before-M2 ordering is correct. Within the declared v1 core scope, however, the plan has two design-level flaws that must be resolved before M1/M2 code exists: (1) sections 3 and 11 describe two contradictory certificate-domain models (cert.domains as authoritative coverage set vs. SAN derived from attached hosts), which leaves every cert lifecycle flow — domain edits, detach, delete-in-use, zero-host certs — undefined; and (2) the apply pipeline validates AFTER overwriting the live include dir, creating a window where a crash/reboot/external reload loads a broken config — the exact catastrophe the pipeline exists to prevent, fixable by validating staging via an alternate root conf before the swap. ## security

**Вердикт/резюме:** The plan shows real security awareness (learns from NPM CVEs: no default creds, no shell interpolation, argon2id, setup-token, non-0.0.0.0 bind) but its central privilege story — "the panel is not root" — is not honest as a security boundary. Because the panel writes files that a root-run Angie master reads and executes, any control over the config content is root-equivalent on the box, and the plan's stated injection defense ("angie -t catches errors before applying") is not an injection control at all: a syntactically VALID injected directive is the attack. Two features punch straight through the boundary: (1) user fields (server_name, forward_host, location path, rewrite) inserted into templates "as-is," and (2) the advanced_snippet escape hatch inserted verbatim. Combined with an HTTP-only LAN panel on a device that is often the network gateway, a missing CORS/DNS-rebinding stance, and a Portainer-style setup-token race, a remote or on-LAN attacker who obtains one authenticated request (CSRF, rebinding, XSS in the diff/stderr views, or winning the setup race) escalates to root code execution as Angie. The fixes are concrete and mostly additive: strict per-field allowlisting, a directive-allowlist validator over the *generated* files (ideally applied by a tiny root helper so the panel never writes http.d directly), gating/allowlisting advanced_snippet, a Host-header allowlist plus explicit CORS deny, a token-required + file-gated setup flow, blocking proxy_pass to loopback management ports, and locking down file perms on the secret-bearing backups/DB. Without the generated-file validator the "not root" claim should be dropped from the docs; with it, the boundary becomes meaningful.

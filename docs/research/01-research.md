# Исследование: Angie ACME, деплой на Armbian, фичи NPM, prior art

> Автогенерировано research-агентами 2026-07-05. Источник фактов — официальные доки/исходники/пакеты.

## angie-acme

**Вердикт/резюме:** Angie (an nginx fork by Web Server LLC) has built-in ACME (RFC 8555) support via the http_acme module (plus a stream ACME module since 1.10.0). Introduced in Angie 1.5.0 (2024-03-27) with http-01 only; dns-01 (including wildcards) and an acme_hook external-hook system arrived in 1.8.0 (2024-12-19); tls-alpn-01 arrived in 1.11.0 (2025-12-24). Newest release as of 2026-07-05 is 1.11.8 (2026-06-18). Configuration centers on `acme_client <name> <ACME-directory-URI> [params]` at http level and `acme <name>;` inside server blocks; certificates are referenced via variables `$acme_cert_<name>` / `$acme_cert_key_<name>` passed to ssl_certificate / ssl_certificate_key. dns-01 is implemented by Angie itself acting as an authoritative DNS responder on UDP port 53 (acme_dns_port): you must NS-delegate _acme-challenge.<domain> to the Angie host — there is no built-in DNS-provider API integration (that use case is covered by acme_hook, which forwards challenge add/remove events to an external handler via fastcgi/proxy/etc.). Wildcard certs require challenge=dns (or a dns hook). Certs/keys/account key are stored under /var/lib/angie/acme/<client_name>/ (account.key, certificate.pem, private.key); renewal is automatic (renew_before_expiry, default 30d; retry after failure default 2h). ACME functionality is identical in OSS and PRO (all ACME features appear in the OSS changelog). External Account Binding (ZeroSSL/Google CA style) is NOT in any released version up to 1.11.8, but an `eab=` parameter (HS256/HS384/HS512) exists in master and is slated for the next release. Key type is ecdsa (P-256) by default, rsa optional.

**Следствия для проекта:**
- A configurator should model ACME as two linked objects: a global (http-level) acme_client entry (name, CA directory URI, challenge type, key_type/key_bits, email, renewal knobs) and a per-server `acme <name>;` reference, then auto-emit ssl_certificate $acme_cert_<name> / ssl_certificate_key $acme_cert_key_<name> — the variable names are derived from the client name, so client names must be validated as identifier-safe and globally unique across http and stream.
- Gate features by target Angie version: acme_client/acme/http-01 >= 1.5.0; dns-01 + wildcards + acme_hook + account_key >= 1.8.0; renew_on_load + hook uri= >= 1.9.0; IP in acme_dns_port >= 1.9.1; stream ACME >= 1.10.0; challenge=alpn + acme_http_port + acme_max_response_size + stats API >= 1.11.0. EAB (eab= parameter) must be gated to 'next release after 1.11.8' / master builds only — do not offer ZeroSSL/Google Trust Services presets for currently released versions.
- Enforce challenge-type constraints in the UI: wildcard server_name entries require challenge=dns (or a dns acme_hook); regex server_name entries must be excluded from the certificate domain list (warn the user); challenge=alpn should warn when the build uses BoringSSL/AWS-LC.
- For dns-01, the configurator must NOT ask for DNS provider API credentials (Angie has no provider integrations) — instead it should generate the required NS + A delegation records for _acme-challenge.<domain> pointing at the Angie host and warn that UDP/53 must be open and that acme_dns_port <= 1024 requires root; provider-API workflows should be offered as an advanced acme_hook + external handler template.
- Always emit a `resolver` directive in the http (and stream) context when any acme_client uses a hostname-based directory URI, otherwise the client silently cannot work; Let's Encrypt production URI https://acme-v02.api.letsencrypt.org/directory is the sensible default, with the staging URI as a testing toggle.
- Since 1.11.0 no dedicated port-80 server block is needed for http-01 (the module creates its own listener via acme_http_port); for versions 1.5.0–1.10.x the configurator must ensure some server block listens on :80.
- Expose sensible defaults matching Angie's own: ECDSA P-256 keys, renew_before_expiry=30d, retry_after_error=2h; offer a dual-cert (RSA+ECDSA) preset by generating two acme_client entries and two acme directives per server (requires >= 1.6.0).
- No OSS/PRO switch is needed for ACME features — the module behaves identically in both editions, so the configurator's ACME UI can be edition-agnostic; optionally surface the /status/http/acme_clients/ monitoring endpoint (>= 1.11.0) for issuance status.
- Storage is self-managed under /var/lib/angie/acme/ (override via acme_client_path); the configurator should surface this path for backup/migration guidance and support the account_key parameter for reusing an existing ACME account.
- Warn users that the first TLS handshakes will fail until the first certificate is issued (the $acme_cert_<name> variable is empty pre-issuance and there is no documented self-signed fallback), and that on issuance failure Angie retries every retry_after_error (2h default) — suggest keeping staging-URI testing before switching to production to avoid Let's Encrypt rate limits.

**Детали:**
- [high] ACME support is provided by the http_acme module (http_acme_module), which implements automatic certificate acquisition per RFC 8555. It is not built by default from source (requires ./configure --with-http_acme_module) but IS included in Angie's official repository packages and Docker images.
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [high] ACME support first appeared in Angie 1.5.0 (released 2024-03-27): 'basic support for automatically obtaining and updating certificates using the ACME protocol, configurable with the acme_client and acme directives, as well as variables of the form $acme_cert_* and $acme_cert_key_*'. At introduction only http-01 was supported.
  - src: https://github.com/webserver-llc/angie/blob/master/CHANGES
- [high] ACME version timeline: 1.5.1 (2024-05-16) reuses stored expired/mismatched certs during renewal + segfault fix; 1.6.0 (2024-06-28) multiple `acme` directives per server block (e.g. dual RSA+ECDSA certs) and BoringSSL support in the ACME module; 1.8.0 (2024-12-19) DNS-01 challenges (incl. wildcard certs), acme_hook hooks system, account_key parameter, extra logging; 1.9.0 (2025-04-11) renew_on_load parameter, uri= parameter for acme_hook, enabled=off semantics change; 1.9.1 (2025-05-29) IP[:port] in acme_dns_port, wildcard+third-level-domain bugfix; 1.10.0 (2025-07-03) ACME in the stream module (acme directive + $acme_cert_* in stream); 1.10.3 (2025-11-13) many stream-ACME bugfixes; 1.11.0 (2025-12-24) tls-alpn-01 (challenge=alpn), acme_http_port directive (no dedicated port-80 server block needed anymore), acme_max_response_size directive replacing max_cert_size for response limit, automatic certificate-size calculation, /status/http/acme_clients/ statistics API section with Prometheus support; 1.11.1 (2025-12-30) acme_http_port binds only to IPs of existing listen directives on that port; 1.11.7 (2026-06-15) fix: DNS-challenge response could be sent from wrong IP/interface; 1.11.8 (2026-06-18) fix: IP without port in acme_http_port/acme_dns_port crashed master process.
  - src: https://github.com/webserver-llc/angie/blob/master/CHANGES
- [high] Newest Angie release as of 2026-07-05 is 1.11.8, released 2026-06-18 (both OSS and PRO carry the same version number and date).
  - src: https://en.angie.software/angie/docs/oss_changes/
- [high] Directive `acme_client name uri [parameters];` (context: http) defines an ACME client with a globally unique name; each client manages exactly one certificate. Parameters: enabled=on|off (default on), key_type=ecdsa|rsa (default ecdsa), key_bits (default 256 for ecdsa / 2048 for rsa), email=, max_cert_size (auto-calculated since 1.11.0), max_key_auth_size (default 2k), renew_before_expiry= (default 30d), renew_on_load (since 1.9.0), retry_after_error= (default 2h; 'off' disables retries), challenge=http|dns|alpn (default http), account_key= (path to existing PEM account key, since 1.8.0). A `resolver` must be configured in the same context for acme_client to work (unless the ACME server URI uses an IP).
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [high] Directive `acme name;` (context: server; also stream server since 1.10.0) attaches an ACME client to a server block. One certificate covers ALL non-regex server_name entries of all server blocks referencing that client; regex server_names are unsupported and skipped. Can be specified multiple times per server block (since 1.6.0) to obtain multiple certificate types.
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [high] Certificates are referenced via built-in variables: `ssl_certificate $acme_cert_<name>; ssl_certificate_key $acme_cert_key_<name>;`. The key variable is available immediately after startup; the certificate variable becomes available only after the client has obtained at least one certificate. Canonical minimal example: acme_client example https://acme-v02.api.letsencrypt.org/directory; server { listen 443 ssl; server_name example.com www.example.com; acme example; ssl_certificate $acme_cert_example; ssl_certificate_key $acme_cert_key_example; }
  - src: https://en.angie.software/angie/docs/configuration/acme/
- [high] Supported challenge types: http-01 (challenge=http, default; since 1.5.0), dns-01 (challenge=dns; since 1.8.0), tls-alpn-01 (challenge=alpn; since 1.11.0, requires an SSL library with ALPN support — not available with BoringSSL/AWS-LC), plus hook-based validation via acme_hook (since 1.8.0) for http and dns challenges handled by an external application.
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [high] dns-01 mechanism: Angie itself acts as the authoritative DNS responder — it answers the ACME server's DNS TXT queries on UDP port 53 (configurable via `acme_dns_port port|ip[:port]`, IP support since 1.9.1; ports <=1024 need superuser). This requires NS delegation in your zone, e.g. `_acme-challenge.example.com. 60 IN NS ns.example.com.` and `ns.example.com. 60 IN A <angie-ip>`, and the Angie host reachable from the internet on UDP/53. Angie does NOT integrate with DNS provider APIs natively; provider-API integration is done via the acme_hook mechanism with an external script/service.
  - src: https://en.angie.software/angie/docs/configuration/acme/
- [high] Wildcard certificates (e.g. *.example.com) are supported since 1.8.0 but ONLY with challenge=dns (or a dns hook); http-01 and tls-alpn-01 cannot issue wildcards. The `acme` docs explicitly state wildcard domains require challenge=dns.
  - src: https://en.angie.software/angie/docs/configuration/acme/
- [high] http-01 mechanics: before 1.11.0 a server block listening on port 80 had to exist; since 1.11.0 the module creates a dedicated listener itself, port configurable via `acme_http_port` (default 80, added 1.11.0, behavior refined 1.11.1: with existing server blocks on that port it binds only their listen IPs).
  - src: https://github.com/webserver-llc/angie/blob/master/CHANGES
- [high] Hook-based validation: `acme_hook name [uri=...];` (context: location; uri= with variables since 1.9.0) makes Angie issue internal subrequests to that location on challenge setup/teardown, exposing variables $acme_hook_name (add|remove), $acme_hook_challenge (dns|http|alpn), $acme_hook_client, $acme_hook_domain (wildcard prefix stripped), $acme_hook_token, $acme_hook_keyauth (TXT record value for dns / response body for http); the location typically fastcgi_pass/proxy_pass to an external handler that e.g. calls a DNS provider API.
  - src: https://en.angie.software/angie/docs/configuration/acme/
- [high] Storage: certificates, private keys and the ACME account key are kept on disk in per-client subdirectories under the path set by the build option --http-acme-client-path, default /var/lib/angie/acme/ in official packages; overridable at runtime with the `acme_client_path` directive (context: http). Files per client: account.key, certificate.pem, private.key. An existing account key can be placed in the client subdirectory or referenced with account_key= to reuse an ACME account.
  - src: https://en.angie.software/angie/docs/configuration/acme/
- [high] Renewal: fully automatic in-process (no cron/external tooling). On startup a stored still-valid certificate is reused instead of re-requesting; renewal is triggered renew_before_expiry (default 30d) before expiration or when the domain list changes; renew_on_load (1.9.0+) forces renewal on config reload; since 1.5.1 expired/mismatched stored certs are still used while renewing. On failure the client retries after retry_after_error (default 2h; off disables).
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [high] OSS vs PRO: there is no ACME feature gap — every ACME feature (dns-01, wildcards, hooks, stream ACME, ALPN, stats API section) appears in the open-source CHANGES file / OSS version history; the PRO changelog lists the same ACME entries with the same version numbers. ACME monitoring is exposed at /status/http/acme_clients/ in the statistics API (since 1.11.0, Prometheus-compatible), which is also available in OSS.
  - src: https://en.angie.software/angie/docs/oss_changes/
- [high] External Account Binding (EAB, needed for ZeroSSL/Google Trust Services/etc.) is NOT supported in any released version up to and including 1.11.8 — no eab parameter exists in the released docs or CHANGES. However, master branch already contains full EAB support: a new `eab=<key-id>:[HS256|HS384|HS512:]<base64url-mac-key>` parameter of acme_client (HS256 default), with parsing in src/http/modules/ngx_http_acme_module.c and a 2026-dated test tests/acme_eab.t stating it is 'a new parameter', so it will land in the next release after 1.11.8.
  - src: https://github.com/webserver-llc/angie/blob/master/tests/acme_eab.t
- [high] Key algorithm choice: key_type=ecdsa (default, key_bits default 256) or key_type=rsa (key_bits default 2048) per acme_client; to serve both RSA and ECDSA certs for one vhost you define two acme_client entries and reference both via two `acme` directives in the server block (supported since 1.6.0).
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [medium] Known limitations/behaviors: regex server_name entries are skipped from certificate requests; 'no valid domain name defined for ACME client' errors if a referencing server block has no ACME-compliant name (since 1.9.0 only when NO valid name exists); enabled=off (since 1.9.0) disables only renewal while keeping variables usable; DNS propagation for NS delegation can take up to 48h; Let's Encrypt rate limits apply; ALPN challenge unavailable with BoringSSL/AWS-LC builds; acme_max_response_size (default 32k, 1.11.0+) may need raising for very large cert chains ('too big subrequest response' error); before the first successful issuance $acme_cert_<name> is unavailable so TLS for that vhost cannot serve a real cert until issuance completes (docs do not describe a temporary/self-signed fallback).
  - src: https://en.angie.software/angie/docs/configuration/modules/http/http_acme/
- [high] Multiple ACME clients in one configuration are fully supported (a crash with 5+ clients was fixed in 1.6.x era per changelog; each client has its own directory URI, so different CAs — e.g. Let's Encrypt production and staging — can be mixed in one config). Each acme_client name must be globally unique across http and stream contexts; using a client only in stream (without http) was fixed to work in 1.10.3.
  - src: https://github.com/webserver-llc/angie/blob/master/CHANGES
- [high] Stream (TCP/UDP) ACME since 1.10.0 (2025-07-03): the `acme` directive and $acme_cert_*/$acme_cert_key_* variables work in stream server blocks for automatic TLS on stream proxies; several interaction bugs (stream-only client disabled with 'defined but not used', stream-before-http ordering emerg, unknown-variable errors) were fixed in 1.10.3.
  - src: https://en.angie.software/news/releases/angie-1-10-0/
- [high] Canonical documentation URLs: module reference https://en.angie.software/angie/docs/configuration/modules/http/http_acme/ ; how-to guide https://en.angie.software/angie/docs/configuration/acme/ ; version history https://en.angie.software/angie/docs/oss_changes/ (angie.software/en/... URLs 302-redirect to en.angie.software).
  - src: https://en.angie.software/angie/docs/configuration/acme/

## angie-deploy

**Вердикт/резюме:** Angie (nginx fork by ex-nginx core devs, current stable 1.11.x as of mid-2026) is fully installable as a native systemd service on Armbian arm64 from the official APT repo: Debian 11/12/13 (bullseye/bookworm/trixie) and Ubuntu 22.04/24.04/26.04 are published for x86-64 and arm64 only — there are NO armhf/armv7 packages, so 32-bit ARM boards need a source build. Armbian keeps Debian's os-release fields (ID=debian, VERSION_ID, VERSION_CODENAME), so the documented repo one-liner works unmodified. The package installs /etc/angie/angie.conf (docs push an include /etc/angie/http.d/*.conf split), unit angie.service, logs in /var/log/angie/; config workflow is angie -t to validate then systemctl reload angie (or angie -s reload → SIGHUP, graceful with automatic rollback on bad config). The repo also carries dynamic-module packages (angie-module-njs, -image-filter, -perl, -xslt, -wasm/-wamr/-wasmtime) and angie-console-light, a purely static real-time monitoring SPA served by Angie itself. The built-in HTTP /status API (api directive) is JSON, GET/HEAD-only in OSS Angie, exposing server info, connections, slabs, resolvers, per-zone HTTP/stream metrics, caches, upstream peer health and ACME client state (zones must be declared via status_zone/zone); api_config_files on additionally exposes the contents of all loaded config files. There is NO write/configuration API in OSS Angie — runtime upstream editing via /config/ (and the Console Light editing UI on top of it) is Angie PRO only — so any web configurator must edit files on disk and trigger a reload. Bonus: native Prometheus export via the prometheus directive + prometheus_all.conf template.

**Следствия для проекта:**
- Native install on Armbian is viable only on arm64 boards: gate the configurator's install flow on `dpkg --print-architecture` == arm64 and Debian 11/12/13 codename; on armhf/armv7 boards the official repo has nothing, so either refuse or offer a from-source build path.
- The official repo one-liner (using $ID/$VERSION_ID $VERSION_CODENAME from /etc/os-release) works unmodified on Armbian because Armbian preserves Debian's ID/VERSION fields — the installer can reuse the documented commands verbatim, but should still verify the resolved URL exists as a sanity check for derivative images that mangle os-release.
- All configuration writes MUST go through the filesystem: OSS Angie has no write API at all, so the configurator's apply pipeline is: write generated files into /etc/angie/http.d/ (and stream.d), run `angie -t` (capture stderr for user-facing validation errors), then `systemctl reload angie` — reload is safe because Angie rolls back to the old config if the new one fails to apply.
- Design the file layout around the packaged include structure: keep /etc/angie/angie.conf untouched and own only files under /etc/angie/http.d/*.conf (one file per vhost/feature), which matches both the shipped package config and official docs guidance.
- For live status in the configurator UI, enable a localhost-restricted `location { api /status/; }` block and poll the JSON tree (connections, server_zones, upstreams with per-peer health, caches, ACME clients); the configurator must auto-inject status_zone / zone directives into generated configs or the metrics simply won't exist.
- `api_config_files on;` gives the configurator a read-back channel: it can fetch the exact config file contents Angie actually loaded via /status/angie/ and diff them against its own generated files to detect out-of-band edits.
- Do not plan any runtime upstream manipulation (add/remove backends without reload) — that is Angie PRO's /config/ API; on OSS the equivalent is rewriting the upstream block file + graceful reload, which is cheap and connection-safe anyway.
- angie-console-light is a free, zero-footprint add-on worth offering as an optional checkbox: it complements (does not compete with) a configurator, since on OSS it is monitoring-only; its documented location block (alias /usr/share/angie-console-light/html/ + /console/api/ -> api /status/) can be templated directly, with allow/deny and auth_basic guarding.
- ACME/Let's Encrypt status is exposed at /status/http/acme_clients/ — Angie has built-in ACME, so the configurator can drive certificates natively instead of shelling out to certbot, and surface renewal state from the API.
- For observability beyond the built-in UI, a single `location =/p8s { prometheus all; }` plus `include prometheus_all.conf;` gives Prometheus metrics with no exporter process — attractive on low-RAM SBCs.
- On low-RAM boards, generated configs should default to conservative nginx-style tuning (worker_processes auto or 1, modest worker_connections); no Angie-specific SBC caveats were found in official sources, and Angie inherits nginx's small event-driven footprint.
- Version note for docs/testing: target Angie 1.11.x (1.11.8 as of June 2026); /status/angie has included build_time since 1.9.0, useful as a capability probe when the configurator talks to older installs.

## prior-art

**Вердикт/резюме:** Prior-art scan of web UIs for nginx/Angie (excluding nginx-proxy-manager). The closest and most mature prior art is nginx-ui (0xJacky): Go backend + Vue 3 frontend, ~11.3k stars, actively released (v2.4.1, 2026-07-03), runs bare-metal via install script + systemd, does raw/block config editing, its OWN embedded ACME client (go-acme/lego, DNS-01 with 100+ providers), monitoring, cluster sync, web terminal. It nominally supports Angie, but only shallowly — since v2.0.0-beta.9 it lets you override the binary path and test/reload/restart commands (issue #191); it does not touch Angie's API or built-in ACME module. NPMplus (ZoeyVid) is a Docker-only Node/TypeScript NPM fork focused on hardened proxy-hosting (HTTP/3, CrowdSec, open-appsec), no Angie support. nginxWebUI (cym1102) is a Java Spring Boot + SQLite structured config generator using acme.sh, deployable as a bare jar. Zoraxy is a Go standalone proxy with UI (not nginx-based) — relevant only as competing UX. On the Angie side, the ONLY existing UI is the official Angie Console Light: a Preact-based, Apache-2.0, monitoring-only panel driven by Angie's REST /status API (config-file viewing via api_config_files; upstream editing only against Angie PRO's /config API). No open-source "web configurator for Angie" exists (GitHub search surfaces only angie-console-light; ispmanager formally rejected Angie support in 2024), and NO existing tool integrates Angie's built-in ACME module — every nginx UI ships its own ACME client (lego/certbot/acme.sh). Angie's ACME module has existed since 1.5.0 (2024-03), gained stream-context support in 1.10.0 (2025-07), and in 1.11.0 (2025-12) gained ALPN challenge support plus an ACME-client statistics API endpoint /status/http/acme_clients/ — i.e., the building blocks for an ACME-native configurator exist in the server itself and are unexploited by any UI.

**Следствия для проекта:**
- Genuine gap confirmed: there is no Angie-native web configurator. The only Angie UI (Console Light) is read-mostly monitoring, and the only tool that 'supports' Angie (nginx-ui) just swaps the binary path and reload commands. An Angie-first configurator that writes config AND exploits Angie's API + built-in ACME has no direct competitor.
- Biggest differentiator vs all nginx UIs: do NOT embed an ACME client. nginx-ui bundles lego, NPMplus bundles certbot, nginxWebUI bundles acme.sh — each duplicating cert state outside the server. With Angie >= 1.5.0 the configurator only needs to GENERATE acme_client/acme directives and reference $acme_cert_<name>/$acme_cert_key_<name> variables; the server does issuance/renewal itself. Target Angie >= 1.11.0 to read ACME client status from /status/http/acme_clients/ for the UI's certificate dashboard (and note the module is present in official packages by default).
- Use Angie's REST /status API (and api_config_files) as the monitoring/observability backbone instead of building agents or parsing logs — this is exactly what Console Light does and it eliminates most of nginx-ui's monitoring code. Console Light is Apache-2.0 (permissive), so its API-consumption patterns can be studied or reused; nginx-ui is AGPL-3.0, so its code cannot be lifted into a non-AGPL project.
- Architecture pattern proven by nginx-ui (the market leader at 11.3k stars): single Go-style binary, bare-metal install script + systemd unit, Docker as an option not a requirement. NPMplus's Docker-only model is a recurring complaint niche — bare-metal/systemd support is a real differentiator and matches Angie's own package-based distribution culture.
- Copy the config-safety workflow every serious tool uses: write to a staging copy -> 'angie -t' validate -> atomic write -> 'angie -s reload' (or systemctl reload angie), with automatic versioned backups and diff/restore (nginx-ui does backup+diff; users expect it).
- Offer BOTH structured editing (proxy-host/site forms like NPM/nginxWebUI for the 80% case) and a raw editor escape hatch (like nginx-ui's Ace/block editor) — tools that only do one lose users at the boundary. For Angie, structured models can additionally surface Angie-specific directives (acme, http/3, upstream 'sticky', resolver zones) that nginx UIs will never model.
- Dynamic (no-reload) upstream editing via the /config API is Angie PRO-only; an OSS-targeted configurator must treat upstream changes as config-file edits + reload, but can feature-detect PRO and light up dynamic editing when /config is available (same pattern Console Light uses).
- Frontend precedent: nginx-ui uses Vue 3, Console Light uses Preact — nothing in this space uses React/shadcn yet, so a React/shadcn frontend is unopposed but also gets no reusable components from prior art; the reusable asset is Angie's well-documented JSON /status schema.
- Naming/discoverability: GitHub searches for an Angie panel currently return almost nothing — an early open-source release could own the 'angie ui / angie configurator' keyword space the way 0xJacky owns 'nginx ui'.
